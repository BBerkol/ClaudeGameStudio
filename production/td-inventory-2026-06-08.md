## Why combat-demo-backlog felt like progress but wasn't

The combat-demo backlog (killed today) had every appearance of forward motion. It was scoped tight ("1 player vs 3 enemies polished sequence"), demo-able, bug-driven, and produced clean commits (32 EditMode test files, ~440 lines of dead authoring deleted, ADR-0010 slot retirement merged as 80b0cbe). The polish captures read like ADR-quality work. The problem is the work shape, not the work quality.

**Failure mode 1: subsystem perfectionism in isolation.** The combat view is the only end-to-end thing playable today, so every "next problem worth solving" surfaces inside it. The Dredge armor bar bug, the card-disappear bug, the VehicleBarStack runtime-spawn refactor, the Layer 2 → Layer 3 animation cut — each is a real problem with a real fix, but each one is *bounded by the only system that exists*. The work auto-narrows to where you can see, which is combat view. Nothing in the planning surface — backlog, polish captures, ADRs — pushes the eye outward toward Node Map / Save / Loot / Economy. The polish captures explicitly serve "the final-game model" but only of the system being polished.

**Failure mode 2: polish refactors are concrete in a way new systems are not.** A "ship Layer 3 hand animation cut" has visible before/after, an eye-check protocol, a measurable diff. "Ship the smallest playable run loop" has no comparable demo. The brain reaches for the work shape with the clearest definition of done. Combat-view polish wins that contest every time, especially solo.

**Failure mode 3: bridge avoidance discipline rewarded the wrong thing.** ADR-0011 (no-bridges) and ADR-0010 (slot vocabulary retirement) were the right architectural calls. But "no bridges anywhere" raises the bar on every single new system, and the natural response to "high bar on new system" is "let's finish polishing the one I have first." The discipline that was supposed to prevent debt accidentally rationalised staying in the comfortable system.

**Failure mode 4: the "shipping the canonical shape later" rationalisation.** Every polish capture this week justified its scope against "this is the final-game shape, we ship it forward." That's true at the file level (the 5-component hand pipeline IS the final shape). But "this file is the final shape" doesn't compose into "we are shipping the final game." The composition into a playable run is the work that wasn't being done.

**Pattern-match for future sessions.** If a polish capture, refactor proposal, or backlog item meets all of: (a) bounded to one existing system, (b) demo-able in isolation, (c) justified as "ships forward to final game," (d) no dependency on a paper MVP system going from paper to built — it is likely the same failure mode. The corrective question: *"what end-to-end run-loop step does this unblock?"* If the answer is "none, but it makes the existing step nicer," the work is polish, not progress.

The combat view will need this work again. It will not need it before Milestone 1 unless Milestone 1 surfaces a specific gap. Defer.
