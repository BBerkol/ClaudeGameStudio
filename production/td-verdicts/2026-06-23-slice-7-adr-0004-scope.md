# TD Verdict — Slice 7 Scope: ADR-0004 First Implementation

**Date:** 2026-06-23
**Gate:** TD-FEASIBILITY (slice scope binding)
**Verdict line:** `TD-FEASIBILITY: CONCERNS`

## Verdict

`TD-FEASIBILITY: CONCERNS` on the current framing ("land two flags into per-system DTO surface"). The framing materially under-describes the work. Reframe Slice 7 explicitly as **"first ADR-0004 landing,"** then ship it as a single canonical slice (Option A, scoped narrow). Option B is an ADR-0011 violation in disguise. Option C is acceptable as a sequencing choice but does not eliminate the debt — it only postpones it.

## Findings (verified against `Wasteland Run/Assets/Scripts/Run/`)

1. **Zero Save infrastructure shipped.** No `IRunStateSerializable`, no `SaveSystem`, no `WastelandRun.Save.Dtos` namespace, no Newtonsoft package in `Packages/manifest.json`, no `link.xml`. Slice 7 is the **first** ADR-0004 landing, not an extension.
2. **The "two flags" framing is stale.** `BiomeDistributionSO` today ships `AllowBidirectional`, `ReconnectRadius` (was `MaxHopDistance` during Block 3), and `MaxEdgeLength`. None of them need DTO surface: `AllowBidirectional` is captured by `BiomeId` snapshot (regen-on-load); the two radii are SO-resident and re-read from disk. Treating these as "what Slice 7 is about" buries the actual scope.
3. **Two xmldoc claims encode unbuilt persistence shape:**
   - `NodeMap.cs:13-19` declares persistence shape `BiomeId + RunSeed + CurrentIndex + PathHistory[]`. `PathHistory[]` does not exist on `NodeMap`. `BiomeId` does not exist on `BiomeDistributionSO`.
   - `RunState.cs:14-20` declares `PendingCardOffer` "survives serialise/reload" — the write-gate (`RunSession.Advance` block) is already live; the writer is not.
4. **"Slice 7" label collision.** `NodeMap.cs:39` calls Slice 7 = forward-pressure/fuel-storm layer. `active.md` calls Slice 7 = ADR-0004 persistence. The codebase and session log disagree. **Fix the label in the same slice that lands the work — don't ship under a name that already means something else in source.**

## Answers to your five questions

### 1. Is Option B a real slice or an ADR-0011 violation?

**Violation.** A `RunStateDto` POCO with no `SaveSystem` reading or writing it is the textbook "stub returns" + "transitional comments" anti-pattern from ADR-0011. The interface exists for a consumer that doesn't exist; the file's reason-to-be is "we'll wire it later." That is precisely what 2026-05-31's no-bridges directive forbids.

**However:** the *data-shape decisions inside it* (`BiomeId` enum on `BiomeDistributionSO`, `PathHistory[]` on `NodeMap`) are **not** bridges. They are runtime-net-additive. `BiomeId` makes Biome 1 self-identifying; `PathHistory` makes the run's traversal queryable (already useful for HUD breadcrumbs, victory screen recap, debug overlays). These can ship without any DTO class and serve real present-day consumers. Split the shape decisions from the DTO. The smell is the empty DTO, not the shape.

### 2. Option A vs C — ordering recommendation

**Run C first, then A.** Reasoning:

- **A is high-stakes and needs full focus.** Background `Task` writes, atomic temp-then-rename, IL2CPP `link.xml`, Mono `File.Move`/`Flush` runtime verification (ADR-0004 OQ4/OQ5), Steam Cloud separation, and 5 CI tests (`SchemaRegistry_Unique_test`, `InterfaceExclusion_test`, `ComputeEnvelopeChecksum_fixture_test`, plus per-DTO round-trip and recovery-chain integration). This is 3+ sessions of careful work. Stapling it to UI Toolkit port debt would be the same mistake as Slice 6's mega-slice.
- **C is bounded and known.** UI Toolkit port of `CombatOutcomeOverlay` + `CardRewardPicker` is concrete ADR-0014 debt with no design questions open. Closing it before A: (a) drains ADR-0014's open debt before opening a new ADR's debt; (b) puts Save's eventual "save indicator widget" surface on UI Toolkit ground (ADR-0004's Negative #3 forward-flags this); (c) honors the user's pacing preference for surgical slices.
- **C is not demo-forward.** Closing UGUI→UIT debt is canonical 1.0 work — it's eliminating a bridge, not building one.

### 3. The `MasteryState` question

**Do not declare `IMasteryStateSerializable` at Slice 7.** ADR-0004 Decision 2 says they're disjoint and CI-enforces it via `InterfaceExclusion_test`. The exclusion test enforces **"no type implements both"** — it does not require both interfaces exist to be meaningful. A reflection test that finds zero offenders is a green test.

Declaring an interface with zero implementers is the "unused abstraction" smell *unless* you can name today's consumer. Mastery isn't in MVP scope I can find. Land `IMasteryStateSerializable` in the slice that lands the first Mastery DTO, not before. The exclusion test code can be authored at Slice 7 and stays green until Mastery arrives.

### 4. The `Advance`-block-shipped-ahead-of-writer question

**Debt to flag, not OK as-is.** The `RunSession.Advance` block exists today (`RunSession.cs:77-79`) and the xmldoc cites ADR-0004 as its justification — but the "survives serialise/reload" half of that justification is fictional until Slice 7 lands. The block is currently sound because `PendingCardOffer` is plain runtime state and the gate makes turn-flow legal — but the **ADR-0004 citation in the xmldoc is a forward-dated promise being collected on**, not a description of present reality.

**Action at Slice 7:** when Save lands, the xmldoc claim becomes accurate, no code change needed. **Action right now:** none — the gate is functionally correct without persistence. But this is exactly the pattern Memory `feedback_capture_before_destroy_view_layer.md` warns about: source claims encoding plans not yet shipped. Track it.

### 5. Recommended slice boundary

**Recommended sequence:**

- **Slice 7a (UI Toolkit port).** `CombatOutcomeOverlay` + `CardRewardPicker` → UXML/USS/C# controller per ADR-0014. Closes existing debt. 1 session.
- **Slice 7b (shape pre-work — net-additive, no DTO class).** Three surgical adds:
  1. `BiomeId` enum (or stable string const) on `BiomeDistributionSO`. `Biome1Distribution.asset` baked as `Biome1`.
  2. `PathHistory: IReadOnlyList<int>` on `NodeMap`, appended-to inside `Advance` (already where `CurrentIndex` mutates). Wire one current consumer to prove it's not stub work (suggestion: MapView breadcrumb trail rendering, or debug overlay).
  3. **Rename memo** — change `Slice 7` references in `NodeMap.cs:39` xmldoc to "forward-pressure layer (future slice)" so the label collision dies. Adjust `active.md` to call the next slice "Slice 8 = ADR-0004."
  No DTO class. No persistence. Ships in 1 session. ADR-0011 clean because every line has a present-day consumer.
- **Slice 8 (the actual ADR-0004 first landing).** Canonical 1.0 shape, narrow:
  - `IRunStateSerializable` interface
  - `RunStateDto` (RunSeed, BiomeId, CurrentIndex, PathHistory, Scrap, Vehicle, Deck, PendingCardOffer, Status) — POCO in `WastelandRun.Save.Dtos`
  - `SaveSystem` public API per ADR-0004 (enqueue + load + helpers + commit/handler predicates)
  - Atomic write (background `Task` + temp-in-`temporaryCachePath` + `File.Move(..., overwrite:true)` + sharing-violation retry budget)
  - Per-category recovery chain (RunState only — Mastery deferred)
  - Newtonsoft package + `link.xml` + IL2CPP smoke test
  - 3 CI tests (`SchemaRegistry_Unique_test`, `InterfaceExclusion_test`, `ComputeEnvelopeChecksum_fixture_test`) + integration test (write → kill → recover via `InjectFault`)
  - Validates OQ4 (`FlushFileBuffers` Mono) + OQ5 (`File.Move` 3-arg overload) per ADR-0004 Risks. **These must be code-review closed in this slice.**
  - `MasteryState`-side: **none.** No interface, no DTO, no second category. Land with Mastery system itself.
  - **Out of scope** for Slice 8: UI surface (save indicator, error toast, blocking dialog), `Project Identity` CI check (separate hygiene task), dev tooling (`InjectFault` is implementation-private at this stage, not console-exposed).
  - Estimate: 2-3 focused sessions.

This sequence:

- Respects ADR-0011: every artifact has a present-day consumer (Slice 7b shape pre-work) or a present-day reader (Slice 8 = SaveSystem reads what it writes, atomically).
- Respects demo-forward retraction: Slice 8 is canonical 1.0 shape, not scaffolding. Newtonsoft and `link.xml` land for real.
- Respects user's pacing: three surgical-sized slices instead of one mega-slice.
- Kills the Slice 7 label collision in 7b before it becomes archaeology.
- Closes ADR-0014 debt before opening ADR-0004 debt (one open ADR at a time on the view layer).

## Success criteria (we'll know this was right if)

- Slice 7b touches `BiomeDistributionSO`, `NodeMap`, one consumer of `PathHistory`. No new namespaces, no new interfaces, no DTO files. Net-additive only.
- Slice 8 lands with zero stubs in `WastelandRun.Save`. Every method has a caller in the same slice. `IMasteryStateSerializable` is absent until Mastery ships.
- `NodeMap.cs:39` xmldoc no longer mentions "Slice 7" by the end of 7b.
- `RunState.cs:14-20` xmldoc reads accurately by the end of Slice 8 (the persistence layer it cites exists).
- Grep for `TODO.*Save` or `// Slice 7` in `Assets/Scripts/Run/` returns zero hits at end of Slice 8.

## What I'd reject

- Option B as proposed (empty DTO class, no SaveSystem). ADR-0011 violation.
- Adding `IMasteryStateSerializable` "for the exclusion test." Unused abstraction. The test goes green with zero implementers.
- Bundling UI Toolkit port into A. Two open ADRs simultaneously on view-adjacent work is the same mistake we just avoided on scrap-readout.
- Stapling `Project Identity` tech-prefs / Steamworks Auto-Cloud config into Slice 8. Out of scope; it's a separate ship-prep slice closer to EA.
